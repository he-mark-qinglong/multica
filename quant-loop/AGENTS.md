# AGENTS.md — quant-loop workspace

> Canonical rule for every curator / quant / strategy agent operating under
> `~/multica/quant-loop/`. Read this **before** answering any
> "do we have data for X?" question, before any data-completeness audit,
> and before any backtest that assumes certain (symbol × timeframe) coverage.

---

## 1. Audit-by-replication rule (mandatory)

### Why this rule exists
`SMA-34855` ran a data-completeness audit that only checked
`~/multica/quant-loop/data/` and `~/multica/quant-loop/live_data/`. It
reported **"no 1m data exists anywhere at workspace level"** — a false
negative. Reality: the 1m bars live under per-strategy local copies
(`strategies/*/data/`) and under `data/perp_1m/` (which the audit missed
entirely). Re-running with the same naive directory list reproduces the bug.

### The rule
**Before drawing any "this symbol/timeframe exists / is missing" conclusion,
enumerate every data file in the workspace first.** The enumeration is the
single source of truth for the matrix. Build the matrix **from the enumeration
result only**, never from a hard-coded directory list.

```bash
# Step 1 — enumerate (mandatory, do not skip, do not paraphrase).
find ~/multica/quant-loop -path '*data*' -type f \
  -not -path '*/__pycache__/*' \
  -not -path '*/.pytest_cache/*' \
  -not -name '*.pyc' \
  -not -name '*.py' \
  -not -name '*.json' \
  -not -name '*.md' \
  -not -name '*.sha256' \
  -not -name '*.ipynb' \
  > /tmp/quant_loop_data_files.txt

# Step 2 — verify the count matches the canonical 234+ files
# (shared pool 30 + freqtrade user_data 4 + strategy-local copies 200+).
wc -l /tmp/quant_loop_data_files.txt
```

If the count is materially different from the canonical snapshot below,
**stop and report the discrepancy** before drawing any conclusions — a
changed count means data was added, removed, or the workspace moved.

**Canonical count as of 2026-07-19:** 253 data files (33 shared perp OHLCV
+ 10 funding + 4 freqtrade user_data + ~206 strategy-local copies across
65 strategies). Re-run periodically; the count grows when new strategies
land.

### Step 3 — classify each file into one of three buckets
The workspace has **three distinct data locations**. An audit that
collapses them into one will mis-report coverage.

| Bucket | Path pattern | Authority | Notes |
|--------|--------------|-----------|-------|
| **Shared pool — perp OHLCV** | `data/perp_{1m,2h,30m}/` and `live_data/` | All strategies | Canonical coverage claims must reference these paths. Currently 33 files. |
| **Shared pool — funding** | `data/funding/` | All strategies | Schema differs (`fundingTime` / `fundingRate`, not `open_time`). |
| **Freqtrade user data** | `freqtrade_v10/user_data/data/` | freqtrade only | Feather + parquet duplicates; framework-local. |
| **Strategy-local copies** | `strategies/*/data/` | Single strategy | Snapshot frozen at strategy-build time. May diverge from shared pool. **Must be reported separately** in any audit; never silently merged with the shared pool. |

### Step 4 — required outputs from any audit
Every data-completeness audit MUST produce, at minimum:

1. **The exact `find` command used** (copy-pasteable, reproducible).
2. **The total file count** the command returned.
3. **A (symbol × timeframe) matrix** with **two rows per cell** — one for
   shared pool (`live_data/` + `data/perp_*`), one for strategy-local
   copies (`strategies/*/data/`). Strategies that happen to have the
   same symbol/timeframe as the shared pool are NOT a substitute for
   the shared pool — they may be stale snapshots.
4. **A "verified vs inferred" column per cell** — verified means the
   audit actually opened the file and counted rows; inferred means the
   audit assumed content from the filename alone.
5. **Symlink audit** — run `ls -la` on `live_data/*.parquet` and
   `readlink -f` to expose any symlinks (the SMA-34855 ETH/SOL 4h bug
   was a symlink that secretly pointed at BTCUSD bars).

### Step 5 — language rule
Never write "no data exists anywhere" without first quoting the
`find` output that backs the claim. The same `find` invocation must be
re-runnable on a different day and produce the same answer.

## 3. Anti-patterns (audit failure modes)

| Anti-pattern | What it produces | Fix |
|--------------|------------------|-----|
| Hard-coded directory list (`ls data/ live_data/`) | False negative — misses 1m/5m/d data | Always `find` first |
| Reporting strategy-local copies as "shared pool coverage" | Over-counting — local copies may be stale | Report shared vs local as separate rows |
| Skipping the symlink audit | False positives (BTCUSD bars read as BTCUSDT) | Always `ls -la` + `readlink -f` |
| Inferring "no file" without quoting the `find` line | Unverifiable claims | Paste the command and the output |
| Reporting file count without distinguishing `.parquet` from `.csv` | Confusing coverage maps | Note both extensions explicitly |
